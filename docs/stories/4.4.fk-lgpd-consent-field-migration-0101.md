## Dev Notes

### Contexto da Story

Esta story fecha o único débito remanescente do EPIC-004. O campo `feedback_data_sharing_consent_by_user_id` registra qual usuário deu consentimento ao compartilhamento de dados de feedback (LGPD). Sem FK, o banco aceita IDs de usuários que não existem — risco de compliance em auditoria LGPD.

O débito foi rastreado como D05 em `docs/db/DB-AUDIT.md` e estava no escopo original do Bloco B do EPIC-004. Foi explicitamente adiado na Story 4.2 (Dev Notes, seção "Out of scope"), pois a Story 4.1 já modificava `companies.ts`. Após a conclusão das 3 stories originais, Quinn (@qa) identificou D05 como follow-up obrigatório no QA final.

[Source: docs/stories/4.2.fks-criticas-indices-performance-migration-0099.md#dev-notes]
[Source: docs/db/DB-AUDIT.md#D05]

### Campo atual em `companies.ts` (linha 29)

```ts
feedbackDataSharingConsentByUserId: text("feedback_data_sharing_consent_by_user_id"),
```

O campo é nullable (sem `.notNull()`), o que é correto — nem toda company tem consentimento registrado. A FK deve usar `ON DELETE SET NULL` para preservar registros históricos de consentimento mesmo que o usuário seja deletado (dado de auditoria LGPD).

[Source: packages/db/src/schema/companies.ts linha 29 — verificado]

### Padrão de ON DELETE para este campo

| Campo | ON DELETE | Justificativa |
|-------|-----------|---------------|
| `feedback_data_sharing_consent_by_user_id` | SET NULL | Dado de auditoria LGPD deve ser preservado; FK aponta para quem deu consentimento, não para o conteúdo em si |

Este mesmo padrão (SET NULL em campos de auditoria) foi usado em `approvals.decided_by_user_id` e `approvals.requested_by_user_id` na Story 4.2.

[Source: docs/stories/4.2.fks-criticas-indices-performance-migration-0099.md#dev-notes]

Reutiliza o import já existente da Story 4.1 (`owner_user_id`). Sem circular dependency, sem necessidade de `AnyPgColumn`. Padrão coerente.

```sql
FOREIGN KEY ("feedback_data_sharing_consent_by_user_id") REFERENCES "public"."user"("id") ON DELETE set null ON UPDATE no action;
```
Constraint nomeada corretamente, referência à tabela `user` correta, `ON UPDATE no action` adequado.
