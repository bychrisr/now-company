## Story

**As a** time de produto do Paperclip,
**I want** uma suíte de testes E2E + isolamento multi-tenant cobrindo todo o fluxo do EPIC-001 (Super Admin → Empresa conecta conta → Sync funciona → Desconexão limpa),
**so that** eu tenha confiança de que nenhuma empresa enxerga dados de outra e o fluxo completo funciona em produção sem regressões.

## Acceptance Criteria

### Cobertura E2E (Playwright)
1. Teste E2E: Super Admin loga, vai em `/instance/settings/platforms`, vê lista completa, desabilita uma plataforma, refresh confirma persistência.
2. Teste E2E: Plataforma desabilitada pelo Super Admin NÃO aparece em "Disponíveis para conectar" no fluxo de empresa.
3. Teste E2E: Empresa loga, vai em `/company/settings/social-accounts`, clica "Conectar" no Instagram, é redirecionada para mock OAuth, callback retorna, conta aparece na lista.
4. Teste E2E: "Sincronizar agora" atualiza `last_synced_at` visível na UI.
5. Teste E2E: "Desconectar" remove conta da lista ativa (após confirmação no modal).

### Isolamento Multi-Tenant (Vitest + DB embedded)
6. Test: Empresa A conecta Instagram (account_id=123). Empresa B faz GET `/api/company/social-accounts` → NÃO vê a conta da empresa A.
7. Test: Empresa A tem `secret_id` X em `company_secrets`. Empresa B tenta acessar diretamente → 403/404 (validar via `company_id` no contexto).
8. Test: Empresa A dispara `POST /sync/:id` da conta da empresa B → 403/404.
9. Test: Deletar empresa A cascateia delete em `company_social_accounts` E marca `company_secrets` como revoked (verificar comportamento esperado em `secret_access_events`).
10. Test: Routine de sync da empresa A processa APENAS contas com `company_id = A`. Conta de empresa B não é tocada.

### Smoke + CI
11. Suíte E2E roda em CI sem flakiness (mock de Instagram OAuth via test fixture).
12. Suíte de isolamento integra ao `pnpm run test` existente.
13. Documentação curta em `tests/e2e/social-accounts-README.md` explicando como rodar localmente (env vars necessárias, mocks).

### Validação final do EPIC-001
14. Todos os Success Criteria do `epic.md` (EPIC-001) validados manualmente em smoke checklist.
15. `pnpm run typecheck`, `pnpm run build`, `pnpm run test`, `pnpm run test:e2e` (ou comando equivalente) passam.

## Dev Notes

### Por que esta story existe
EPIC-001 toca infraestrutura crítica (multi-tenant, secrets, OAuth). Sem suíte de isolamento explícita, qualquer regressão futura pode vazar dados entre empresas — risco crítico.

### Cobertura mínima de isolamento
Toda nova feature multi-tenant deve ter, no mínimo:
1. Empresa A não lê dados de B (GET)
2. Empresa A não modifica dados de B (PATCH/POST/DELETE)
3. Empresa A não executa ações em recursos de B (sync/trigger)
4. Cascade de delete não afeta dados de outras empresas

Esta story estabelece o padrão para futuras stories do EPIC-002 e EPIC-003.

### Padrões a seguir
- E2E: Playwright (verificar setup existente em `tests/e2e/`)
- Integration tests: Vitest com DB embedded (padrão do projeto)
- Mocks OAuth: NÃO bater na API real em CI
- Fixtures: localizadas em `tests/e2e/fixtures/`

### Out of scope (não esta story)
- ❌ Testes de performance/load → fase futura
- ❌ Testes de outras plataformas além de Instagram → seguem o padrão estabelecido aqui
- ❌ Testes de UI estilizada (snapshot visual) → fase futura

### Testing
- **Não criar mocks parciais** — se mock Instagram, mockar TUDO (auth + metrics endpoints)
- **Test data isolado** — cada test cria suas próprias empresas, não compartilha estado
- **Cleanup obrigatório** — afterEach limpa tabelas afetadas
