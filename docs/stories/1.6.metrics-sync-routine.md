## Story

**As a** empresa do Paperclip,
**I want** que as métricas das minhas contas sociais (follower_count, avg_engagement_rate) sejam sincronizadas automaticamente via cron,
**so that** o dashboard mostre dados atualizados sem que eu precise clicar em "atualizar" manualmente.

## Acceptance Criteria

1. Existe Routine cadastrada (via `routines` table existente) para sync de métricas de contas sociais.
2. Routine tem `routine_triggers` com cron expression (padrão sugerido: `0 */6 * * *` — a cada 6 horas).
3. Routine é por empresa (`company_id`) — cada empresa tem sua própria Routine independente.
4. Execução da Routine itera por `company_social_accounts` `WHERE company_id = X AND is_active = true`.
5. Para cada conta, executa fetch na API da plataforma (Instagram Graph API nesta story) usando o token do `company_secrets` referenciado por `secret_id`.
6. Atualiza campos `follower_count`, `avg_engagement_rate`, `last_synced_at` na tabela `company_social_accounts`. Edge cases: `avg_engagement_rate = NULL` quando `followers_count = 0` ou quando a conta não tem posts retornados.
7. Erros de sync por conta são registrados em `routine_runs.failureReason` (para a routine como um todo, quando houver erro agregado) e em `logger.error` estruturado por conta com `{ companyId, accountId, platformSlug, errorCode, errorMessage }`. Falha de uma conta NÃO impede o sync das demais (error isolation por conta).
8. Token expirado (401/invalid_token) dispara tentativa de refresh via Instagram `GET /refresh_access_token`. Pré-requisito: token salvo precisa ser long-lived (60d) — corrigir Story 1.5 para trocar short-lived por long-lived após exchange. Falha de refresh marca conta como `needs_reauth = true` (campo dedicado, ver Task 3).
9. Rate limit da plataforma respeitado — implementar backoff exponencial em caso de 429.
10. Endpoint manual `POST /api/company/social-accounts/:id/sync` permite trigger manual da sync para uma conta específica (útil para debug e UX).
11. `pnpm run typecheck`, `pnpm run build`, `pnpm run test` passam.

### Approach

1. **Scheduler design (decisão chave):** scheduler separado de `tickScheduledTriggers` para evitar acoplamento com o pipeline de heartbeat (que dispacha runs para agents AI). Usei `kind = "social_metrics_sync"` no `routine_triggers` para isolar — `tickScheduledTriggers` filtra apenas `kind = "schedule"`, sem conflito.

2. **System agent placeholder:** `agents.companyId` é NOT NULL, então não há "system agent global". Solução: lazy-create por empresa via `ensureMetricsSyncRoutine` (status=paused, adapter=process, role=system). Apenas placeholder pra satisfazer FK em `routines.assigneeAgentId` — nunca executa runtime.

3. **Error isolation em 2 camadas:** (a) `syncMetricsForCompany` itera contas com try/catch defensivo extra; (b) `syncOneAccount` captura todos os erros internamente e retorna `SyncAccountResult` em vez de throw. Garante que uma conta quebrada nunca impede as outras.

4. **Token refresh:** detectado por `InstagramApiError.kind === "auth"` (401/403). Tenta refresh; em sucesso, rotaciona o secret via `secretService.rotate` e re-tenta sync. Falha de refresh → `needs_reauth=true` + `sync_error="token_refresh_failed"`.

5. **Backoff exponencial:** wrapper `fetchWithBackoff` faz retry interno em 429 com delays 1s→2s→4s→8s→16s→32s (cap 60s, max 5 retries). Testável via injeção de `sleep`.

6. **Edge cases AC6:** `computeAvgEngagementRate` retorna `null` se `followers ≤ 0` ou `media.length === 0`. Armazenado como `NULL` em `avg_engagement_rate` (coluna nullable).

7. **Fix oauth-callback (Story 1.5):** adicionado `exchangeForLongLivedToken` que troca short-lived (1h) por long-lived (60d) via `/access_token?grant_type=ig_exchange_token`. Refresh só funciona em long-lived, então este fix é pré-requisito de AC8.

### Issues por Severidade

#### 🟢 LOW — Race condition em `ensureMetricsSyncRoutine`
Sem unique constraint em `routines(companyId, title)` ou `agents(companyId, name)`. Duas chamadas concorrentes (ex: 2 logins OAuth simultâneos) podem criar duplicatas.
**Impacto:** baixo (cenário raro; duplicatas não quebram funcionalidade — apenas geram lixo).
**Recomendação:** adicionar unique index em story futura OU usar `INSERT ... ON CONFLICT DO NOTHING`.

#### 🟢 LOW — Test 2 (error isolation) depende de ordem de chamadas
Mock alterna respostas por contador, não por token. Funciona mas é frágil se a ordem de iteração mudar.
**Impacto:** nenhum em produção; apenas qualidade do teste.
**Recomendação:** refatorar mock pra rotear por header `Authorization` em story futura.

#### 🟢 LOW — AC10: POST `/:id/sync` sem teste dedicado
Cobertura indireta via `syncOneAccount` (testado), mas a route em si (auth + 404 + binding) não tem teste de integração.
**Impacto:** baixo — auth helpers são reutilizados e já testados em `social-accounts.test.ts`.
**Recomendação:** adicionar 2 testes (401 sem auth, 404 conta inexistente) em sprint futuro.

#### 🟢 LOW — POST `/sync` não restringe à plataforma Instagram
A route aceita qualquer `account.id`, mas `syncOneAccount` só funciona pra Instagram (URLs hardcoded). Para uma conta de outra plataforma, retornaria erro genérico.
**Impacto:** baixo — só Instagram existe por enquanto.
**Recomendação:** quando adicionar 2ª plataforma (YouTube, etc.), refatorar para abstração por plataforma.

#### 🟢 LOW — Sem reaper para `routine_runs` "running" travados
Se o servidor crashar mid-sync, o `routine_run` fica em `status='running'` para sempre.
**Impacto:** baixo — não bloqueia próximos runs; só polui o histórico.
**Recomendação:** reaper simples (>15min em running → status=failed) em story de observabilidade futura.
