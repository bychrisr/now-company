## Story

**As a** Super Admin (usuário com role elevado em `instance_user_roles`),
**I want** uma interface em `/instance/settings/platforms` para visualizar, habilitar e desabilitar plataformas do catálogo global,
**so that** eu controle quais plataformas estão disponíveis para todas as empresas da instância sem precisar manipular o banco diretamente.

## Acceptance Criteria

1. Existe rota nova em `server/src/routes/` (ex: `social-platforms-admin.ts`) com endpoints REST:
   - `GET /api/instance/platforms` — lista todas as plataformas (Super Admin only)
   - `PATCH /api/instance/platforms/:id` — atualiza `status` (`enabled`/`disabled`) e `sort_order`
2. Todos os endpoints são protegidos por middleware equivalente a `assertCanManageInstanceSettings()` (consultar `server/src/services/access.ts` para o padrão exato existente). Request sem permissão retorna 403.
3. Rota registrada em `server/src/routes/instance-settings.ts` (ou no router pai que agrupa rotas de instance).
4. Existe nova página em `ui/src/pages/InstancePlatformsAdmin.tsx` (ou nome equivalente, seguindo padrão de outras páginas em `ui/src/pages/`).
5. Página é acessível somente para Super Admin — usuário comum não vê o item no `InstanceSidebar` e tentativa de acesso direto via URL retorna 403/redirect.
6. Item "Plataformas" adicionado em `ui/src/components/InstanceSidebar.tsx` apontando para a nova rota (condicional a `isInstanceAdmin`).
7. UI lista todas as plataformas com colunas: ícone (se houver), slug, name, category, status (toggle enabled/disabled).
8. Toggle de status persiste imediatamente via `PATCH` — UI mostra feedback de loading/erro.
9. UI segue padrão visual de `ui/src/pages/AdapterManager.tsx` (mesma estrutura de cards/tabela/header).
10. `pnpm run typecheck` passa sem erros.
11. `pnpm run build` passa sem erros.
12. `pnpm run test` passa (incluindo eventual `InstancePlatformsAdmin.test.tsx` se houver padrão de teste para páginas).

### Padrões a seguir

- **Backend:**
  - Rotas REST com Zod para validação de body/params
  - Middleware de auth ANTES do handler
  - Erros estruturados (consultar `server/src/errors.ts` para padrão)
- **Frontend:**
  - Componentes funcionais com hooks
  - Estado servidor via lib que o projeto usa (verificar — provavelmente React Query/TanStack Query)
  - Tailwind 4.0 + tokens OKLCH em uso (per CLAUDE.md)

### Decisões fixas (não eliciar)

- **Sem CRUD completo**: esta story só permite habilitar/desabilitar + sort_order. Criar/deletar plataformas é Super Admin power-user feature, fica para fase futura. Plataformas existem via seed (Stories 1.1 e 1.3).
- **Sem edição de specs**: campos `capabilities/copy_specs/image_specs` não são editáveis pela UI nesta story. São gerenciados via seed/migration (Story 1.3 popula).

### Out of scope (vai em outras stories)

- ❌ CRUD completo (create/delete) → fase futura, não previsto
- ❌ Edição de specs técnicos via UI → não previsto
- ❌ Upload de ícone → fase futura
- ❌ Tabela `company_social_accounts` → Story 1.4
- ❌ Seed expandido com 15+ plataformas → Story 1.3
