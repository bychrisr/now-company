## Story

**As a** empresa do Paperclip,
**I want** uma tabela `company_social_accounts` para registrar minhas contas conectadas em cada plataforma social,
**so that** eu possa publicar conteúdo via OAuth, sincronizar métricas e cross-postar — sem que tokens fiquem expostos fora do cofre.

## Acceptance Criteria

1. Existe migration `0095_*.sql` em `packages/db/src/migrations/` criando a tabela `company_social_accounts`.
2. Existe arquivo `packages/db/src/schema/company_social_accounts.ts` definindo a tabela com Drizzle ORM.
3. Schema exportado em `packages/db/src/schema/index.ts`.
4. Campos obrigatórios da tabela:
   - `id` UUID PK
   - `company_id` UUID FK → `companies.id` (cascade on delete)
   - `platform_id` UUID FK → `social_platforms.id` (restrict on delete)
   - `secret_id` UUID FK NULLABLE → `company_secrets.id` (set null on delete) — armazena OAuth tokens
   - `handle` text NOT NULL — ex: "@paperclipai"
   - `display_name` text
   - `profile_url` text
   - `platform_account_id` text — ID da conta na plataforma (Instagram Graph ID, YouTube channel ID, etc.)
   - `follower_count` integer NOT NULL default 0
   - `avg_engagement_rate` numeric(5,4) — ex: 0.0345 = 3.45%
   - `last_synced_at` timestamptz NULLABLE
   - `default_hashtags` text[] default '{}'
   - `default_cta` text
   - `timezone` text default 'America/Sao_Paulo'
   - `is_active` boolean NOT NULL default true
   - `is_verified` boolean NOT NULL default false
   - `created_at`, `updated_at` timestamptz NOT NULL default now()
5. Unique constraint: `(company_id, platform_id, platform_account_id)` — empresa não conecta a mesma conta 2x.
6. Índices:
   - `(company_id, is_active)` — listar contas ativas da empresa
   - `(company_id, platform_id)` — agrupar por plataforma
   - `secret_id` (parcial onde NOT NULL) — lookup pelo secret
7. **NUNCA armazenar `access_token` ou `refresh_token` diretamente na tabela** — apenas via `secret_id` → `company_secrets`.
8. `pnpm run db:migrate`, `pnpm run typecheck`, `pnpm run build` passam.

## Dev Notes

### Decisão registrada
`docs/plans/content-brand-module-brainstorm.md` Bloco 4A (2026-05-30): "A tabela NÃO guarda `access_token`/`refresh_token` diretamente. Guarda apenas `secret_id` (FK → `company_secrets`)."

### Padrão de FKs
- `companies.id` → cascade (deletou empresa, some todas as contas)
- `social_platforms.id` → restrict (não deletar plataforma com contas conectadas)
- `company_secrets.id` → set null (se secret for revogado, conta fica órfã mas não some)
