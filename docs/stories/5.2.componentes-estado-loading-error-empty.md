## Story

**As a** usuário do Paperclip,  
**I want** que todas as páginas mostrem estados de carregamento, erro e vazio padronizados,  
**so that** eu nunca veja uma tela em branco ou silêncio sem feedback — sempre saberei o que está acontecendo com o sistema.

## Acceptance Criteria

1. Componente `<ErrorState>` é criado em `ui/src/components/ErrorState.tsx` com props: `message: string`, `retry?: () => void`, `details?: string` — renderiza mensagem de erro, detalhe opcional e botão de retry
2. Componente `<EmptyState>` existente em `ui/src/components/EmptyState.tsx` é ampliado: props `action` e `onAction` passam a aceitar `ReactNode` além de `string`, mantendo backward compatibility com callers existentes
3. O ícone `LucideIcon` em `EmptyState` recebe `aria-hidden={true}` (corrige problema de acessibilidade F13 parcial)
4. `<ErrorState>` é aplicado nas seguintes páginas/componentes, **substituindo** os padrões inline de `text-destructive`:
   - `ui/src/pages/Agents.tsx` (AgentList — estado de erro de listagem)
   - `ui/src/components/IssuesList.tsx` (IssueList — estado de erro de listagem)
   - `ui/src/pages/Costs.tsx` (CostDashboard — erro de fetch de custos)
   - `ui/src/pages/Approvals.tsx` (ApprovalsQueue — erro de fetch de aprovações)
   - `ui/src/pages/CompanySettings.tsx` (erro de fetch/update de settings)
5. `<EmptyState>` já aplicado ou sem estado vazio é verificado nos 6 alvos — onde ausente, o componente existente é aplicado com ícone, mensagem e CTA adequados
6. Toast notifications em `ui/src/components/ToastViewport.tsx` recebem `role="status"` (corrige F15 — acessibilidade de toasts)
7. `pnpm test:run` passa sem regressões — testes existentes de `Agents`, `IssuesList`, `CompanySettings` devem continuar passando
8. `pnpm build` completa sem warnings
9. Nenhuma prop existente de `EmptyState` é quebrada — todos os callers atuais continuam funcionando sem modificação

## Dev Notes

### Contexto da Story

Esta story cria e padroniza os estados fundamentais de UI que estão ausentes ou inconsistentes em 6 páginas críticas. O padrão atual usa inline `text-destructive` sem retry, sem estrutura consistente. [Source: docs/frontend/frontend-spec.md#7.2]

### Componentes existentes relevantes

**`EmptyState.tsx`** já existe em `ui/src/components/`:
```tsx
// Props atuais:
interface EmptyStateProps {
  icon: LucideIcon;
  message: string;
  action?: string;
  onAction?: () => void;
}
// Problema: action é só string, ícone sem aria-hidden
```
[Source: ui/src/components/EmptyState.tsx verificado]

**`PageSkeleton.tsx`** já existe com 8 variantes (list, issues-list, detail, dashboard, approvals, costs, inbox, org-chart). Esta story NÃO adiciona variantes — usa as existentes.
[Source: ui/src/components/PageSkeleton.tsx verificado]

**`ErrorState`** NÃO existe — deve ser criado do zero.
[Source: frontend-spec.md#7.2 — "sem ErrorState ou ErrorBoundary em lugar algum"]

### Design System — tokens disponíveis

```css
/* Para ErrorState usar: */
--destructive       /* cor de erro */
--muted-foreground  /* texto secundário */
--muted             /* background sutil */
--border            /* bordas */
```
[Source: docs/frontend/frontend-spec.md#2.2]

Design majoritariamente flat — `--radius: 0`. Não adicionar border-radius nos novos componentes a não ser que use `--radius-sm`.

### Stack React — padrões existentes

- **TanStack Query:** `useQuery` retorna `{ data, isLoading, isError, error, refetch }` — usar esse padrão nos 6 alvos
- **Tailwind v4:** `@theme inline` + variáveis CSS — usar tokens via `bg-muted`, `text-destructive`, etc.
- **shadcn/ui Button:** `<Button variant="outline" size="sm">` para retry action
- **Lucide icons:** importar individualmente (`import { AlertCircle } from "lucide-react"`)

### Callers existentes de EmptyState (não quebrar)

Verificar via grep antes de alterar props:
```bash
grep -r "EmptyState" ui/src --include="*.tsx" -l
```

### Decisões fixas (não eliciar)

- **Sem ErrorBoundary:** Esta story não implementa Error Boundary (React class component) — apenas o componente `<ErrorState>` funcional para estados de fetch error
- **Sem design system storybook:** Projeto não tem Storybook — validação é visual no app rodando
- **Backward compatible:** `EmptyState` não pode quebrar nenhum caller existente

### Out of scope

- ❌ Loading state dos 6 alvos (PageSkeleton já existe e está aplicado — verificar na Task 1, mas não é foco desta story)
- ❌ ErrorBoundary global → backlog técnico separado
- ❌ Empty state de `RunHistory` — componente não localizado no mapeamento inicial; verificar na Task 1
