## Story

**As a** desenvolvedor implementando o Personal Mode (ORION),  
**I want** que a tabela `companies` tenha os campos `kind` e `owner_user_id`,  
**so that** seja possível distinguir entre companies pessoais e de time, e associar um dono explícito a companies pessoais para controle de acesso e billing.

## Acceptance Criteria

1. A coluna `kind text NOT NULL DEFAULT 'business'` é adicionada à tabela `companies` via migration `0098`
2. A coluna `owner_user_id text REFERENCES "user"(id) ON DELETE SET NULL` é adicionada à tabela `companies` via migration `0098`
3. Um índice único parcial `CREATE UNIQUE INDEX companies_personal_owner_idx ON companies(owner_user_id) WHERE kind = 'personal'` é criado — garantindo que cada usuário tenha no máximo 1 company pessoal
4. O arquivo `packages/db/src/schema/companies.ts` é atualizado com os dois novos campos em Drizzle ORM
5. O rollback plan é documentado no Change Log desta story: `ALTER TABLE companies DROP COLUMN kind; ALTER TABLE companies DROP COLUMN owner_user_id;`
6. `pnpm -r typecheck` passa sem erros após a atualização do schema Drizzle
7. `pnpm test:run` passa sem regressões
8. `pnpm db:generate` gera corretamente a migration SQL a partir do schema Drizzle atualizado
9. Rows existentes em `companies` recebem `kind = 'business'` como valor default (backward compatibility garantida)
10. `docs/db/DB-AUDIT.md` é atualizado: D29 e D30 marcados como `✅ Resolvido`

## Dev Notes

### Contexto da Story

Esta é a story **P0 bloqueadora** do EPIC-004. Sem `companies.kind`, o Personal Mode (ORION) não pode distinguir companies pessoais de workspaces de time. Sem `companies.owner_user_id`, não há como associar uma company pessoal a um dono para controle de acesso e billing.

[Source: docs/stories/epics/epic-004-tech-debt-db-schema/epic.md#scope]

### Schema atual de `companies` (packages/db/src/schema/companies.ts)

```ts
// Arquivo atual — campos relevantes:
export const companies = pgTable(
  "companies",
  {
    id: uuid("id").primaryKey().defaultRandom(),
    name: text("name").notNull(),
    status: text("status").notNull().default("active"),
    // ... outros campos ...
    createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
    updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
  },
  (table) => ({
    issuePrefixUniqueIdx: uniqueIndex("companies_issue_prefix_idx").on(table.issuePrefix),
  }),
);
```

**Não existem** `kind` nem `owner_user_id` atualmente. [Source: packages/db/src/schema/companies.ts verificado]

### Tabela `user` — referência da FK

A tabela `user` usa `id: text` (não UUID) — conforme padrão Better Auth. Portanto `owner_user_id` deve ser `text`, não `uuid`.

```ts
// packages/db/src/schema/auth.ts
export const users = pgTable("user", {
  id: text("id").primaryKey(),
  // ...
});
```

[Source: packages/db/src/schema/auth.ts verificado]

### Risco: circular dependency entre companies.ts e auth.ts

Se `auth.ts` já importa de `companies.ts`, importar `user` em `companies.ts` cria circular dependency. Verificar antes de adicionar o import:

```bash
grep -r "import.*companies" packages/db/src/schema/auth.ts
```

Se houver circular, usar referência lazy do Drizzle:
```ts
ownerUserId: text("owner_user_id").references((): AnyPgColumn => user.id, { onDelete: "set null" }),
```

O padrão de referência lazy já existe em `heartbeat_runs.ts` com o import `AnyPgColumn`. [Source: packages/db/src/schema/heartbeat_runs.ts linha 1]

### Índice único parcial — sintaxe Drizzle

Para índice condicional (WHERE clause) em Drizzle:

```ts
// Requer import de { sql } from "drizzle-orm"
personalOwnerUniqueIdx: uniqueIndex("companies_personal_owner_idx")
  .on(table.ownerUserId)
  .where(sql`kind = 'personal'`),
```

### Decisões fixas (não eliciar)

- **`kind DEFAULT 'business'`:** Obrigatório para backward compatibility com rows existentes
- **`owner_user_id` nullable:** Apenas companies do tipo `personal` terão owner — companies de time não precisam
- **ON DELETE SET NULL:** Se o usuário dono for deletado, a company pessoal não é deletada mas perde o owner
