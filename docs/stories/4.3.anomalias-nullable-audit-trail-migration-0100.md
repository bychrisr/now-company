## Story

**As a** desenvolvedor garantindo a consistência arquitetural do schema,  
**I want** que a tabela `verification` tenha `created_at` e `updated_at` como `NOT NULL`,  
**so that** o schema seja consistente com todos os outros 96 tabelas do projeto que têm esses campos como `NOT NULL`, eliminando a única anomalia de nullability de timestamps no banco.

## Acceptance Criteria

1. **Verificação pré-migration:** Query confirma que não existem rows em `verification` com `created_at IS NULL` ou `updated_at IS NULL` — resultado registrado no Change Log
2. `ALTER TABLE verification ALTER COLUMN created_at SET NOT NULL` é executado via migration `0100`
3. `ALTER TABLE verification ALTER COLUMN updated_at SET NOT NULL` é executado via migration `0100`
4. O schema Drizzle em `packages/db/src/schema/auth.ts` é atualizado: `authVerifications` passa a ter `.notNull().defaultNow()` nos dois campos de timestamp
5. `pnpm db:generate` gera a migration `0100` com os dois `ALTER COLUMN ... SET NOT NULL`
6. `pnpm -r typecheck` passa sem erros após atualização do schema
7. `pnpm test:run` passa sem regressões
8. `docs/db/DB-AUDIT.md` atualizado: D35 marcado como `✅ Resolvido`
9. Rollback plan documentado no Change Log: `ALTER TABLE verification ALTER COLUMN created_at DROP NOT NULL; ALTER TABLE verification ALTER COLUMN updated_at DROP NOT NULL;`

## Dev Notes

### Contexto da Story

`verification` é a **única tabela** entre as 97 do schema que tem `created_at` e `updated_at` como nullable. Todas as demais tabelas usam `NOT NULL DEFAULT now()`. Esta inconsistência é uma anomalia arquitetural documentada como D35.

A tabela `verification` é gerenciada pelo Better Auth e armazena tokens temporários de verificação de email. Better Auth sempre popula esses campos ao criar registros, mas o schema atual não garante isso a nível de banco.

[Source: docs/db/DB-AUDIT.md#D35, packages/db/src/schema/auth.ts verificado]

### Schema atual da tabela `verification`

```ts
export const authVerifications = pgTable("verification", {
  id: text("id").primaryKey(),
  identifier: text("identifier").notNull(),
  value: text("value").notNull(),
  expiresAt: timestamp("expires_at", { withTimezone: true }).notNull(),
  createdAt: timestamp("created_at", { withTimezone: true }),  // ← nullable (anomalia)
  updatedAt: timestamp("updated_at", { withTimezone: true }),  // ← nullable (anomalia)
});
```

[Source: packages/db/src/schema/auth.ts verificado]

### Atenção: tabela gerenciada pelo Better Auth

Better Auth controla os inserts nesta tabela. Após esta migration, Better Auth continuará funcionando normalmente (ele já populava `created_at` e `updated_at`). A mudança apenas impõe a constraint no banco.

Se houver rows com NULL (improvável, mas possível em dev), o backfill da Task 1 corrige antes de aplicar NOT NULL.

### Padrão do projeto — todos os outros timestamps

```ts
// Padrão em todas as 96 outras tabelas:
createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
```

Esta story apenas alinha `verification` com o padrão universal.

### Rollback é trivial

`ALTER COLUMN ... DROP NOT NULL` não afeta dados — apenas remove a constraint. Rollback é seguro em qualquer momento.

### Decisões fixas (não eliciar)

- **Backfill first:** Se COUNT > 0, fazer UPDATE antes de SET NOT NULL (never fail migration mid-flight)
- **Escopo mínimo:** Esta story toca APENAS `verification`. Outras anomalias de timestamp (D20–D24) ficam em backlog — são documentadas como exceções intencionais (append-only, imutáveis)

### Out of scope

- ❌ D20–D24 (heartbeat_run_events, cost_events, etc.) → exceções intencionais, documentar no schema como comentários (backlog P3)
- ❌ D36, D38 (tokens OAuth em plaintext) → alto esforço, sprint separado
