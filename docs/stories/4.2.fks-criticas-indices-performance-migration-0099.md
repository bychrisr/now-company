## Story

**As a** desenvolvedor mantendo a integridade referencial do banco de dados,  
**I want** que as FKs de `user_id` críticas sejam formalmente declaradas e que índices de alta frequência sejam adicionados,  
**so that** o banco não aceite registros órfãos em tabelas de governança, e queries de listagem de projetos, goals e autenticação usem índices ao invés de full seq scans.

## Acceptance Criteria

1. `instance_user_roles.user_id` recebe FK formal: `REFERENCES "user"(id) ON DELETE CASCADE`
2. `approvals.decided_by_user_id` recebe FK formal: `REFERENCES "user"(id) ON DELETE SET NULL`
3. `approvals.requested_by_user_id` recebe FK formal: `REFERENCES "user"(id) ON DELETE SET NULL`
4. `agent_runtime_state.last_run_id` recebe FK formal: `REFERENCES heartbeat_runs(id) ON DELETE SET NULL`
5. Índice único é criado em `user.email`: `CREATE UNIQUE INDEX user_email_unique_idx ON "user"(email)`
6. Índice compound é criado em `projects(company_id, status)`: `CREATE INDEX projects_company_status_idx ON projects(company_id, status)`
7. Índice compound é criado em `goals(company_id, status)`: `CREATE INDEX goals_company_status_idx ON goals(company_id, status)`
8. **Verificação de dados órfãos ANTES de aplicar cada FK:** Query de diagnóstico é executada e resultado é registrado no Change Log (zero órfãos é pré-condição para aplicar a constraint)
9. Os schemas Drizzle correspondentes são atualizados: `instance_user_roles.ts`, `approvals.ts`, `agent_runtime_state.ts`
10. `pnpm db:generate` gera a migration `0099` com todos os `ADD CONSTRAINT` e `CREATE INDEX`
11. `pnpm -r typecheck` passa sem erros
12. `pnpm test:run` passa sem regressões
13. `docs/db/DB-AUDIT.md` atualizado: D01, D04, D05, D14, D31, D32, D33 marcados como `✅ Resolvido`

## Dev Notes

### Contexto da Story

Esta story endereça 7 débitos de média/alta prioridade em um único batch. A divisão em FKs + índices é intencional: ambos são DDL não-destrutivo (ADD CONSTRAINT e CREATE INDEX são transacionais e fazem rollback automático em caso de falha).

[Source: docs/stories/epics/epic-004-tech-debt-db-schema/epic.md#scope]

### Schema atual dos arquivos a modificar

**`instance_user_roles.ts`** — `userId` sem FK:
```ts
userId: text("user_id").notNull(),  // sem .references()
```
[Source: packages/db/src/schema/instance_user_roles.ts verificado]

**`approvals.ts`** — `requestedByUserId` e `decidedByUserId` sem FK:
```ts
requestedByUserId: text("requested_by_user_id"),     // sem .references()
decidedByUserId: text("decided_by_user_id"),         // sem .references()
```
[Source: packages/db/src/schema/approvals.ts verificado]

**`agent_runtime_state.ts`** — `lastRunId` sem FK:
```ts
lastRunId: uuid("last_run_id"),  // sem .references()
```
[Source: packages/db/src/schema/agent_runtime_state.ts verificado]

### Tabela referenciada: `heartbeat_runs`

A tabela chama-se `heartbeat_runs` (não `agent_heartbeat_runs`). O export Drizzle é `heartbeatRuns`.
[Source: packages/db/src/schema/heartbeat_runs.ts verificado]

### Padrão de ON DELETE para FKs de user_id

| Caso | ON DELETE |
|------|-----------|
| `instance_user_roles.user_id` | CASCADE — registro de role não faz sentido sem o usuário |
| `approvals.decided_by_user_id` | SET NULL — decisão histórica deve ser preservada, mas sem apontar para usuário deletado |
| `approvals.requested_by_user_id` | SET NULL — idem |
| `agent_runtime_state.last_run_id` | SET NULL — estado do agente é mantido, mas referência ao run é apagada |

### Atenção: `auth.ts` já exporta `users`

Confirmar o nome do export em `auth.ts` antes de importar:
```bash
grep "^export const" packages/db/src/schema/auth.ts
```

### Índice em `user.email` — pode haver email duplicado!

Antes de criar `UNIQUE INDEX` em `user.email`, verificar:
```sql
SELECT email, COUNT(*) FROM "user" GROUP BY email HAVING COUNT(*) > 1;
```
Se houver duplicados, a migration falhará. Escalar para @architect.
