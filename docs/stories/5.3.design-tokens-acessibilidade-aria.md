## Story

**As a** desenvolvedor e usuário com necessidades de acessibilidade,  
**I want** que os tokens CSS do design system sejam consolidados em um arquivo canônico e que os problemas críticos de ARIA sejam corrigidos,  
**so that** futuras mudanças de tema sejam feitas em um único lugar, e usuários de screen readers possam navegar as sidebars e avisos sem ambiguidade.

## Acceptance Criteria

1. Arquivo `ui/src/styles/tokens.css` é criado com todos os tokens CSS canônicos documentados (cores, espaçamentos, tipografia, border-radius, tokens especiais de search chips e anotações)
2. O arquivo `ui/src/index.css` é refatorado para importar `tokens.css` em vez de definir os tokens inline — o conteúdo dos tokens se move para `tokens.css`, não é duplicado
3. `pnpm build` completa sem warnings após a refatoração do CSS
4. Nenhum visual regression é introduzido — layout e cores das páginas existentes permanecem idênticos antes e depois da refatoração
5. `<Sidebar>` em `ui/src/components/Sidebar.tsx` recebe `aria-label="Navegação principal"` (corrige ambiguidade de múltiplas `<aside>` na página)
6. `<InstanceSidebar>` em `ui/src/components/InstanceSidebar.tsx` recebe `aria-label="Navegação de instância"`
7. `<CompanySettingsSidebar>` em `ui/src/components/CompanySettingsSidebar.tsx` recebe `aria-label="Navegação de configurações"`
8. Botões icon-only em `ui/src/components/` sem `aria-label` ou `title` visível recebem `aria-label` descritivo — foco nos `IconButton` ou `Button` com apenas ícone Lucide como filho
9. Hierarquia de headings em `CompanySettings.tsx` e `AgentDetail.tsx` é verificada e corrigida se h1→h2→h3 não for respeitada
10. `pnpm test:run` passa sem regressões
11. `pnpm -r typecheck` passa sem erros

### ARIA — problemas identificados no frontend-spec.md

| Problema | Arquivo | Fix |
|---------|---------|-----|
| `<aside>` sem aria-label (múltiplas sidebars) | Sidebar, InstanceSidebar, CompanySettingsSidebar | `aria-label` em cada `<aside>` |
| `EmptyState` ícone sem aria-hidden | EmptyState.tsx (coberto em Story 5.2) | já tratado em 5.2 |
| ToastViewport sem role="status" | ToastViewport.tsx (coberto em Story 5.2) | já tratado em 5.2 |

[Source: docs/frontend/frontend-spec.md#8.2]

**Nota:** `OrgChart` e `KanbanBoard` sem ARIA de grid/tree **ficam fora de escopo** desta story — complexidade maior, impacto visual possível.

### Vite + Tailwind CSS v4 — import de CSS

Tailwind v4 suporta `@import` nativo no CSS. O padrão:
```css
/* index.css */
@import "./styles/tokens.css";
@import "tailwindcss";
```

Verificar a ordem dos imports — tokens devem ser importados antes do `@tailwind` / `@import "tailwindcss"` para que as variáveis sejam resolvidas corretamente.

### Decisões fixas (não eliciar)

- **Não renomear tokens:** Nenhum token CSS pode ser renomeado — apenas movido de arquivo
- **Flat design preservado:** `--radius: 0` é intencional, não é um bug
- **Sem dark mode novo:** Dark mode já existe — esta story apenas refatora tokens existentes

### Out of scope

- ❌ OrgChart ARIA (`role="tree"`) → complexo, risco de regressão visual
- ❌ KanbanBoard ARIA (`role="grid"`) → idem
- ❌ `FoldCurtain` aria-hidden durante transição → animação não crítica
- ❌ Novos tokens (dark mode completo, novos semantic tokens) → sprint de design system separado
